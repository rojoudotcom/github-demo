# GitHub フロー ハンズオン手順

このリポジトリ（`rojoudotcom/github-demo`）を教材に、**GitHub フロー**を一巡り体験します。
GitHub フローとは「`main` を常に動く状態に保ちながら、変更はブランチと Pull Request（PR）で進める」開発の流れです。

- **公開アプリ**: https://rojoudotcom.github.io/github-demo/
- **やること**: ブランチを切る → 変更 → PR を出す → CI（自動テスト）を見る → マージ → 自動デプロイを確認

---

## 全体像

```
main（常に公開・保護されている）
  │
  ├─ ① ブランチを作る    feature/xxx
  │      │
  │      ② 変更してコミット
  │      │
  │      ③ push して PR を作る ──▶ ④ CI が自動で回る（Lint / Unit テスト）
  │                                     │ 緑になったら
  │◀──────────── ⑤ マージ ◀────────────┘
  │
  └─ ⑥ main への push で自動デプロイ ──▶ GitHub Pages が更新される
```

`main` には保護ルールがかかっており、**PR と CI 通過なしにはマージできません**。これが「壊れた変更が公開に流れない」仕組みです。

---

## 事前準備

1. Git が使えること（`git --version` で確認）
2. GitHub アカウントにログイン済みであること
3. リポジトリをローカルに取得する

   ```bash
   git clone https://github.com/rojoudotcom/github-demo.git
   cd github-demo
   ```

> 手元でアプリを動かして確認したい場合（任意・Node.js 20+ が必要）
>
> ```bash
> npm install      # 初回のみ
> npm run serve    # http://localhost:4173 が開く
> ```

## ハンズオン本編

### ① ブランチを作る

`main` から作業用ブランチを切ります。ブランチ名は「何をするか」がわかる名前にします。

```bash
git switch -c feature/change-heading
```

### ② 変更してコミットする

例として、アプリの見出しを変えてみます。`index.html` の次の行を探し、

```html
<h1 class="app__title">タスクボード</h1>
```

好きな文言に書き換えます（例: `本日のやること`）。保存したらコミットします。

```bash
git add index.html
git commit -m "見出しを本日のやることに変更"
```

### ③ push して Pull Request を作る

作業ブランチを GitHub に送ります。

```bash
git push -u origin feature/change-heading
```

push すると、GitHub の画面上部に **「Compare & pull request」** ボタンが出ます。これを押して PR を作成します（タイトルと説明を書いて「Create pull request」）。

### ④ CI（自動テスト）の結果を見る

PR を作ると、下部の **Checks** に2つのジョブが自動で走ります。ここが GitHub Flow の「関所」です。
**両方が緑（✓）** になって初めてマージでき、**どちらか赤（✗）** ならマージボタンは押せません。
赤のときはログを開いて原因を直し、もう一度 push すれば CI が再実行されます。

以下は、それぞれの検査が「何を・なぜ」見ているかの解説です。

#### チェック1: Lint（コード体裁チェック）

**コードを実行せずに**、書き方と体裁を静的に点検する検査です。2つの道具を組み合わせています。

- **ESLint** — バグの芽や危険な書き方を見つけます。たとえば「使っていない変数」（消し忘れ・書き間違いのサイン）や、
  「定義されていない名前の参照」（タイプミスなど）を検出します。
- **Prettier** — インデント・クォート・1行の長さといった**見た目の体裁**を機械的にそろえます。

なぜ必要か: 体裁のブレやうっかりミスを、レビューより前に自動で弾けます。全員のコードが同じ形に整うので、
差分（diff）が「本質的な変更」だけになり、レビューが速くなります。

赤くなったときの直し方: まず `npm run format` で整形を自動修正し、残った指摘（未使用変数など）を手で直してから
push し直します。手元で試すには `npm run lint` を実行します。

#### チェック2: ユニットテスト

`src/taskStore.js` の**ロジック（純粋関数）**が正しく動くかを、**ブラウザを起動せずに**単体で検証する検査です。
Node 標準の仕組み（`node:test`）を使い、ミリ秒で終わります。

「純粋関数」とは、同じ入力なら必ず同じ出力を返し、外の状態を書き換えない関数のことです。このアプリでは
タスクの追加・完了の反転・削除・絞り込み・未完了件数の計算をすべて純粋関数にしてあります。だからこそ
「この入力を渡したら、この結果になるはず」という形で、DOM も保存領域も用意せずにテストできます。

主な確認内容（`test/unit/taskStore.test.js`）:

- `addTask` … 新しいタスクを一意の番号で追加する／前後の空白を除去する／空のタイトルは拒否する
- `toggleTask` … 指定したタスクだけ完了⇄未完了を切り替える
- `deleteTask` … 指定したタスクだけ削除する
- `filterTasks` … 「すべて / 未完了 / 完了」で正しく絞り込む
- `remainingCount` … 未完了の件数だけを数える

なぜ必要か: 画面をポチポチ触って確かめなくても、ロジックの取り違えを一瞬で見つけられます。手元では
`npm test` で同じ検査を再現できます。

> 💡 **わざと失敗させてみる**: `src/taskStore.js` の `remainingCount` の戻り値をわざと変える
> （例: `.length` に `+ 1` を足す）と、ユニットテストが赤くなり、マージがブロックされる様子を体験できます。
> Lint を試すなら、どこかに未使用の変数（例: `const x = 1;`）を足すと Lint が赤くなります。

### ⑤ レビューしてマージする

CI が緑になったら、PR 画面の **「Merge pull request」** を押して `main` に取り込みます。マージ後は作業ブランチを削除して構いません（画面の「Delete branch」）。

### ⑥ 自動デプロイを確認する

`main` に変更が入ると、**Deploy ワークフローが自動で走り、GitHub Pages が更新されます**。

- リポジトリの **Actions** タブで「Deploy to GitHub Pages」の実行を確認
- 完了後、公開URLを再読み込みして変更が反映されているか確認

  https://rojoudotcom.github.io/github-demo/

これで GitHub フローの一巡り（ブランチ → PR → CI → マージ → デプロイ）が完了です。

## よくあるつまずき

| 症状                                        | 原因と対処                                                                       |
| ------------------------------------------- | -------------------------------------------------------------------------------- |
| `main` に直接 push しようとしたら拒否された | 保護ルールで直接 push は禁止。ブランチを切って PR から進める（これが正しい流れ） |
| マージボタンが押せない                      | CI がまだ実行中、または赤い。緑になるまで待つ／原因を直して再 push               |
| Pages に変更が反映されない                  | Deploy ワークフローの完了を待つ（数十秒〜数分）。Actions タブで成否を確認        |

## 用語ミニ辞典

- **ブランチ**: `main` から枝分かれさせた作業用の線。互いに影響せず並行して進められる。
- **Pull Request（PR）**: 「この変更を `main` に入れてほしい」という提案。差分・レビュー・CI 結果が1画面に集まる。
- **CI（継続的インテグレーション）**: 変更のたびに自動でテスト・検査を回す仕組み。ここでは PR ごとに実行。
- **CD（継続的デリバリー／デプロイ）**: 検査を通った変更を自動で公開まで届ける仕組み。ここでは `main` への取り込みで Pages に自動デプロイ。
